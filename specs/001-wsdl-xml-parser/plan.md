# Implementation Plan: WSDL XML Parser

**Branch**: `001-wsdl-xml-parser` | **Date**: 2026-04-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-wsdl-xml-parser/spec.md`

## Summary

Develop the `XsdXmlParser.Core` WSDL 1.1 and XSD parser library for .NET 6.0, .NET 7.0, and .NET 8.0 with strict element-to-class mapping, schema-first design, graph-based processing, a centralized ID-based registry, and clean separation of concerns. The library provides an async-first public API with full dependency injection support via Microsoft.Extensions.DependencyInjection.

**Scale/Scope**: Named element-to-class mapping; full XSD facet support; annotation documentation

## Coding Standards & Static Analysis

### Single Responsibility Principle (SRP)
- Each method shall have a single, well-defined purpose (max 30 lines of code unless complex logic requires)
- Parse logic separated from validation, which is separated from generation
- Use private helper methods to extract responsibilities
- Leverage static analysis tools to enforce method complexity

### Static Analysis Tools
Configure and enforce with CI/CD pipeline:
- **StyleCop.Analyzers**: C# style and consistency rules
- **FxCop (Roslyn)**: Code analysis and design rules
- **SonarAnalyzer**: Code quality, maintainability, security
- **AsyncFixer**: Async/await best practices

### Dependency Injection & DI Container Registration
- All services must be registered with `Microsoft.Extensions.DependencyInjection`
- TypeRegistry as singleton DI service
- Parser services (WSDLParser, XSDParser) registered as scoped
- ElementParser instances managed by TypeRegistry (singleton, stateless)
- Expose public `AddXsdXmlParser()` extension method for `IServiceCollection`
- Consumer applications can register via: `services.AddXsdXmlParser()`
- Service abstractions and implementations expose async-first methods and propagate `CancellationToken`

### Inheritance & Composition
- Create abstract base classes for common element parser behavior
- Use composition to avoid inheritance hierarchies >3 levels deep
- Favor composition for behavior variation (e.g., constraint validators)
- ElementParsers inherit from `BaseElementParser` abstract class

## Architecture Patterns

### Type Registry (Singleton)
- Central mapping of registry IDs, element names, and namespaces to canonical type definitions and ElementParser singletons
- Initialized during schema parsing
- Thread-safe access for concurrent parse operations
- Registered with DI as singleton

### Element Parsers (Stateless Singletons)
- One ElementParser class per named XSD element
- No instance state; parse position/depth passed as method parameters
- Registered in TypeRegistry at schema load time
- Reused across multiple parse operations
- Decorated with composition-based constraint validators

### Service Layer (Scoped)
- WSDLParserService: coordinates asynchronous WSDL file parsing, delegates to XSDParser for embedded schemas, and propagates `CancellationToken`
- XSDParserService: orchestrates asynchronous XSD parsing, uses element parsers from TypeRegistry, and propagates `CancellationToken`
- JSONMetadataGeneratorService: asynchronously traverses the TypeRegistry graph and produces JSON metadata
- XMLGeneratorService: asynchronously generates XML from schema + data and enforces constraints

### Validators (Composition)
- Separate validator classes for each XSD facet type (length, pattern, enumeration, etc.)
- Element parsers inject/compose validators rather than inherit from them
- Validators are stateless, registered as singletons or transients per DI policy

## Complexity Tracking

| Architecture Decision | Justification |
|---|---|
| Stateless singleton ElementParsers in TypeRegistry | Supports schema reuse, reduces memory overhead; parse state passed as method parameters enables parallel parsing |
| BaseElementParser abstract base class | Enforces SRP; centralizes common validation, metadata attachment, and documentation handling; simplifies inheritance tree |
| DI via ServiceCollectionExtensions | Standard .NET Core pattern; simplifies consumer integration; aligns with dependency injection best practices; supports realistic integration verification through composed service resolution |
| Separate ElementParser folder (one class per named element) | Enforces strict one-to-one element-to-class mapping requirement; improves maintainability and testability at scale (10k+ elements) |
| Composition validators over inheritance | More flexible constraint handling; avoids deep inheritance chains; supports dynamic validator attachment |
| Multi-targeted .NET 6.0 + 7.0 + 8.0 | Preserves consumer compatibility across the supported LTS/current matrix while keeping shared code on C# 10.0 |
| Async-first public API with cancellation | Supports scalable I/O-bound parsing workflows, avoids sync-over-async deadlocks, and improves consumer integration in hosted applications |
| Centralized ID-based registry | Enforces DRY for type definitions, enables stable cross-file references, and simplifies schema merge behavior |
